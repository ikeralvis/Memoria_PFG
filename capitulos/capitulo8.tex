\chapter{Capítulo Octavo - Desarrollo del Sistema}
\label{cap:desarrollo_sistema}

Este capítulo recoge la especificación, el diseño y la implementación de TankGo, así como
el plan de pruebas y el manual de usuario. Es el núcleo técnico del proyecto y documenta
las decisiones de arquitectura, las tecnologías empleadas y el resultado final del sistema.

\section{Especificación de Requisitos}
\label{sec:especificacion}
% ═════════════════════════════════════════════

\subsection{Contexto del sistema}
\label{subsec:contexto}

TankGo es una plataforma de inteligencia de datos orientada a la movilidad y la eficiencia
energética. Su propósito es transformar los datos abiertos del Geoportal de Gasolineras del
Ministerio para la Transición Ecológica~\cite{geoportal_gasolineras} en información
accionable para el conductor, añadiendo capas de valor mediante análisis histórico,
predicción de precios y recomendación geoespacial de repostaje.

El sistema actúa como intermediario entre las fuentes de datos públicos y el usuario final,
operando en un entorno cloud que permite tanto la consulta en tiempo real como el
entrenamiento y servicio de modelos de machine learning. A diferencia de un visor estático,
TankGo integra en una única plataforma la consulta de gasolineras y electrolineras, la
estimación de precios futuros y un asistente conversacional por voz para interacción en
manos libres.

\subsection{Identificación de usuarios}
\label{subsec:identificacion_usuarios}

El sistema contempla dos perfiles de usuario diferenciados. El \textbf{usuario no registrado}
puede acceder a las funciones de consulta pública: visualización de gasolineras y
electrolineras en mapa y tabla, detalle de cada estación y precios actuales. El
\textbf{usuario registrado} dispone, además, de la gestión de estaciones favoritas, el
acceso al histórico de precios de cada estación y las predicciones a 7 días generadas por
el modelo de machine learning.

Dentro de estos perfiles, los casos de uso más representativos son el conductor particular
que busca repostar barato en su área, el transportista profesional que planifica rutas de
larga distancia con criterios de coste, y el usuario de vehículo eléctrico que necesita
localizar puntos de recarga en ruta.

\subsection{Requisitos no funcionales}
\label{subsec:requisitos_no_funcionales}

Los requisitos no funcionales establecen las propiedades de calidad del sistema, no
directamente relacionadas con las funciones que ofrece, sino con cómo las
ofrece~\cite{sommerville2011}.

\textbf{Disponibilidad (RNF-01):} Al tratarse de una plataforma de consulta en ruta, el
sistema debe garantizar un tiempo de actividad superior al 99\%, respaldado por la
infraestructura de Google Cloud Platform.

\textbf{Latencia (RNF-02):} El tiempo de respuesta de las consultas de precios y el
asistente conversacional no deben ser muy elevados, requisito crítico para garantizar
la seguridad vial en uso móvil.

\textbf{Escalabilidad (RNF-03):} La arquitectura de microservicios y el despliegue en
la nube permiten escalar cada componente de forma independiente según la demanda, sin
afectar al resto del sistema.

\textbf{Usabilidad (RNF-04):} El frontend está diseñado con criterio \textit{mobile-first}
y es completamente responsivo, priorizando la experiencia en dispositivos móviles sin
renunciar a la usabilidad en escritorio.

\textbf{Seguridad (RNF-05):} Las comunicaciones se realizan sobre HTTPS, la autenticación
se gestiona mediante tokens JWT y el acceso a los microservicios internos está restringido
a través del API Gateway.

\subsection{Restricciones}
\label{subsec:restricciones}

El sistema opera bajo tres restricciones principales. En primer lugar, existe una
\textbf{dependencia de fuentes externas}: los datos de gasolineras provienen de la API
pública del Ministerio para la Transición Ecológica~\cite{datosgob_carburantes}, cuya
disponibilidad y frecuencia de actualización están fuera del control del sistema. En segundo
lugar, las funciones de consulta, predicción y recomendación requieren \textbf{conectividad
activa a internet}, ya que toda la lógica reside en el backend cloud. Finalmente, el
\textbf{asistente conversacional} está diseñado para responder consultas puntuales, no para
mantener diálogos extendidos mientras se conduce, en línea con las directrices de seguridad
vial aplicables. El sistema de recomendacion de rutas tambien añade una dependecia que esta fuera del control del sistema, 
ya que se basa en la API de Open Route Service para el cálculo de rutas y distancias.

\section{Diseño del Sistema}
\label{sec:diseño_sistema}
% ═════════════════════════════════════════════

\subsection{Alternativas exploradas y decisiones tecnológicas}
Las decisiones tecnológicas de TankGo quedan recogidas en detalle en el Capítulo \ref{cap:marco_tecnologico}
(Marco Tecnológico), donde se justifica la elección de cada
componente del stack frente a las alternativas evaluadas. En esta sección
se describen únicamente los aspectos de diseño arquitectónico que
condicionan la implementación.


\subsection{Arquitectura general}

TankGo sigue una \textbf{arquitectura de microservicios} en la que cada funcionalidad
principal del sistema se encapsula en un servicio independiente con su propia lógica y
despliegue. Los servicios se comunican entre sí a través de una capa de API Gateway que
actúa como punto de entrada único para el cliente.

La arquitectura se organiza en cuatro niveles. El \textbf{cliente} es la aplicación React
PWA que el usuario final consume desde el navegador o dispositivo móvil. El
\textbf{API Gateway} (Hono) recibe todas las peticiones del cliente, gestiona la
autenticación y las enruta hacia el microservicio correspondiente. La \textbf{capa de
  servicios} agrupa los cinco microservicios del dominio: usuarios, gasolineras,
electrolineras, recomendación y predicción. Finalmente, la \textbf{capa de persistencia}
centraliza el almacenamiento en PostgreSQL, accesible desde cada servicio según sus
necesidades.

\subsection{Diseño lógico y físico}
Desde el punto de vista \textbf{lógico}, el sistema se estructura en torno a cinco
entidades de dominio principales: \textit{Estación de servicio}, \textit{Precio},
\textit{Usuario}, \textit{Favorito} y \textit{Predicción}. Las estaciones concentran la
información geolocalizada de cada gasolinera o electrolinera, los precios registran tanto
el valor actual como el histórico asociado a cada estación y tipo de combustible, los
usuarios encapsulan la identidad y preferencias de cada cuenta, los favoritos modelan la
relación entre usuario y estación, y las predicciones almacenan las estimaciones generadas
por el modelo de machine learning para su consulta posterior.

Desde el punto de vista \textbf{físico}, cada microservicio se ejecuta de forma
independiente. La orquestación, contenedores y detalles operativos (plataforma de
ejecución, imágenes, pipelines y gestión de secretos) se describen en la sección
\textit{Despliegue en GCP y pipeline CI/CD}.

\subsection{Modelo de datos}

El esquema de base de datos de TankGo se compone de cinco tablas principales cuyo diseño
refleja directamente el dominio del sistema.

La tabla \texttt{gasolineras} es la entidad central del modelo. Almacena la información
estática y los precios actuales de cada estación de servicio, identificada de forma única
mediante el campo \texttt{ideess} (identificador oficial del Ministerio). Incluye datos de
localización (municipio, provincia, dirección, coordenadas y geometría PostGIS), horario
en formato estructurado (\texttt{horario\_parsed} en JSONB) y los precios actuales de cada
tipo de combustible: Gasolina 95 E5, 98 E5, Gasóleo A, Gasóleo B, Gasóleo Premium,
Gasolina 95 Premium y Diésel Renovable.

La tabla \texttt{precios\_historicos} registra la evolución temporal de los precios por
estación. Cada registro asocia un \texttt{ideess} con una fecha y los precios de los
distintos combustibles en ese momento, lo que permite construir series temporales para el
análisis histórico y el entrenamiento del modelo de predicción.

La tabla \texttt{prediction\_forecasts} almacena las estimaciones generadas por el modelo
LightGBM. Su clave primaria es compuesta: \texttt{ideess}, \texttt{fuel},
\texttt{forecast\_date} y \texttt{run\_date}, lo que permite distinguir predicciones para
diferentes combustibles, horizontes temporales y ejecuciones del modelo. Cada registro
incluye el precio predicho y los márgenes de confianza inferior y superior.

La tabla \texttt{users} gestiona las cuentas de usuario, incluyendo autenticación
tradicional mediante \texttt{password\_hash} y autenticación federada mediante
\texttt{google\_id} (OAuth). Almacena también las preferencias del usuario como el
combustible favorito, el modelo de coche y el tipo de combustible del vehículo.

La tabla \texttt{user\_favorites} modela la relación entre usuarios y estaciones,
permitiendo que cada usuario registre sus gasolineras favoritas. Referencia mediante
claves foráneas tanto a \texttt{users.id} como a \texttt{gasolineras.ideess}.

Las dependencias entre tablas son las siguientes: \texttt{precios\_historicos} y
\texttt{user\_favorites} referencian \texttt{gasolineras} a través de \texttt{ideess},
\texttt{prediction\_forecasts} referencia a \texttt{gasolineras} y
\texttt{user\_favorites} referencia a \texttt{users} mediante \texttt{user\_id}. El
diagrama entidad-relación de la figura~\ref{fig:er} ilustra estas relaciones.

\section{Consideraciones sobre la Implementación}
\label{sec:implementacion}
% ═════════════════════════════════════════════

Esta sección documenta las decisiones de implementación más relevantes de cada componente
del sistema, con especial atención a los aspectos que condicionan el comportamiento en
producción y que no quedan completamente cubiertos por la especificación de requisitos o el
diseño arquitectónico.

% ─────────────────────────────────────────────
\subsection{Estructura del proyecto}
\label{subsec:estructura}
% ─────────────────────────────────────────────

El repositorio sigue una estructura de monorepo, cada microservicio reside en su propio
directorio raíz y tiene su propio \texttt{Dockerfile}, su fichero de dependencias y su
conjunto de pruebas. Esta organización permite iterar sobre un servicio sin alterar el
resto y facilita la asignación de un pipeline de CI/CD independiente a cada componente \cite{newman2021building}.

Los directorios principales son \texttt{gateway-hono}, \texttt{gasolineras-service},
\texttt{usuarios-service}, \\
\texttt{recomendacion-service}, \texttt{prediction-service},
\texttt{voice-assistant-service}, \texttt{mcp-gasolineras-server} y
\texttt{frontend-client}. En la raíz se encuentran el \texttt{docker-compose.yml} para el
entorno local y los siete ficheros \texttt{cloudbuild-*.yaml} para el despliegue en
producción.

La orquestación local se basa en \textbf{Docker Compose}~\cite{docker_compose}, con
dependencias de arranque declaradas explícitamente, los servicios esperan a que
la base de datos esté accesible, el gateway espera a que los servicios de dominio superen
sus \textit{health checks}, el frontend arranca únicamente cuando el gateway está
disponible. Los servicios opcionales de IA y predicción se activan mediante perfiles Docker
(\texttt{ai}, \texttt{ml})~\cite{docker_compose_profiles}, de forma que la instalación base no requiere claves de API
externas.

% ─────────────────────────────────────────────
\subsection{Servicio de gasolineras y sincronización de datos}
\label{subsec:gasolineras_impl}
% ─────────────────────────────────────────────

El \texttt{gasolineras-service} es el componente con mayor complejidad operacional del
sistema, ya que combina la gestión de datos en tiempo real con la necesidad de mantener
frescura respecto a una fuente externa que actualiza sus precios varias veces al
día~\cite{datosgob_carburantes}.

\subsubsection*{Ciclo de sincronización}
El servicio mantiene la frescura de los datos mediante tres mecanismos complementarios:

\begin{itemize}
  \item Comprobación al arranque: si \texttt{AUTO\_ENSURE\_FRESH\_ON\_STARTUP} está activa, se
        valida la antigüedad local y se lanza una sincronización condicional.
  \item Job programado: un \textbf{Cloud Scheduler} ejecuta diariamente un Cloud Run Job que
        sincroniza desde la API del Ministerio y exporta un snapshot Parquet a Google Cloud
        Storage para el pipeline de ML.
  \item Sincronización bajo demanda: si una lectura detecta datos obsoletos y
        \texttt{AUTO\_SYNC\_ON\_READ=true}, se puede desencadenar una sincronización en
        segundo plano (en producción esta opción suele estar desactivada para evitar latencia).
\end{itemize}

Estos tres mecanismos combinados mantienen los datos con menos de 24 horas de antigüedad
en producción sin depender de un sistema de colas externo.

\subsubsection*{Almacenamiento y consultas geoespaciales}

Los precios actuales se almacenan en la tabla \texttt{gasolineras} junto con las
coordenadas geográficas de cada estación. Para las búsquedas por proximidad se utiliza
\textbf{PostGIS}~\cite{postgis}, la extensión geoespacial de PostgreSQL, que permite
ejecutar consultas basadas en distancia con los operadores \texttt{ST\_DWithin} y
\texttt{ST\_Distance} de forma eficiente mediante índices espaciales. Esto evita el
cálculo de la distancia haversine en capa de aplicación para el filtrado inicial,
reservándolo únicamente para el refinamiento de resultados.

El histórico de precios se registra en la tabla \texttt{precios\_historicos}, donde cada
fila representa una instantánea de los precios de una estación en una fecha concreta.

\subsubsection*{Fallback en memoria}

Si al arrancar el servicio no dispone de conexión a la base de datos, entra en modo
\textit{memory-fallback}: carga los datos desde la última sincronización disponible en
memoria y continúa respondiendo peticiones, aunque sin capacidad de escritura. Este modo
garantiza la resiliencia ante interrupciones transitorias de conectividad con Neon, que al
ser un servicio externo puede experimentar latencias de conexión en arranques en frío
(\textit{cold starts}) de Cloud Run.

\subsubsection*{Integración de electrolineras}

Los puntos de recarga eléctrica se obtienen de la API de Mapareve~\cite{mitma_esmovilidad}
y están integrados directamente dentro del \texttt{gasolineras-service}, bajo el prefijo
de rutas \texttt{/api/charging/}. Esta decisión evita la proliferación de microservicios
para funcionalidades que comparten el mismo patrón de acceso (consulta geoespacial, caché
corta) y reduce la complejidad operacional del despliegue.

% ─────────────────────────────────────────────
\subsection{Servicio de usuarios y autenticación}
\label{subsec:usuarios_impl}
% ─────────────────────────────────────────────

El \texttt{usuarios-service} gestiona el ciclo completo de identidad: registro,
autenticación tradicional, autenticación federada con Google OAuth y gestión de favoritos.
Está implementado con \textbf{Fastify}~\cite{fastify}, que en combinación con el plugin
\texttt{@fastify/jwt}~\cite{fastify_jwt} proporciona la firma y verificación de tokens sin código boilerplate.

\subsubsection*{Autenticación y tokens JWT}

Tras un inicio de sesión correcto, el servicio genera un token \textbf{JWT}~\cite{jwt_rfc7519}
firmado con el algoritmo HS256 y una caducidad de 7 días. El token no se devuelve en el
cuerpo de la respuesta, sino que el API Gateway lo establece como una \textit{cookie}
\texttt{httpOnly} con los flags \texttt{Secure} y \texttt{SameSite=None} en producción.
Este mecanismo impide que el token sea accesible desde JavaScript del navegador, mitigando
ataques XSS~\cite{owasp_xss}.

Para el flujo de Google OAuth, el gateway actúa como intermediario: recibe el
\texttt{id\_token} de Google, lo verifica y llama a un endpoint interno protegido
(véase la sección \textbf{Seguridad transversal} \ref{subsec:seguridad_transversal}) para realizar el \textit{upsert} del
usuario. A continuación el servicio devuelve el JWT al gateway para su encapsulación
en cookie.

Las contraseñas se almacenan únicamente como hash generado con \textbf{bcrypt}~\cite{bcrypt}
con 12 rondas de sal, lo que hace computacionalmente inviable su recuperación por fuerza
bruta incluso en caso de exfiltración de la base de datos.

\subsubsection*{Gestión del esquema de base de datos}

El proyecto no emplea un sistema de migraciones automático (por ejemplo, Flyway o Liquibase). El esquema se mantiene en un fichero
\texttt{init.sql} con sentencias idempotentes (\texttt{CREATE TABLE IF NOT EXISTS, ALTER TABLE IF NOT EXISTS}) que se aplican manualmente sobre la
instancia PostgreSQL de Neon en el primer despliegue. Esta aproximación es adecuada para el alcance del proyecto, pero en entornos de
producción con varios despliegues y equipos conviene usar un gestor de migraciones formal para controlar la evolución del esquema.

Herramientas como Flyway o Liquibase permiten versionar y aplicar los cambios de forma reproducible. Flyway se basa en scripts SQL
ordenados por versión, lo que facilita su integración en pipelines CI/CD. Liquibase añade además formatos declarativos (XML/JSON/YAML)
y funciones avanzadas de rollback y auditoría. En proyectos con múltiples entornos estas soluciones reducen el riesgo de desincronización
del esquema y permiten desplegar cambios de forma trazable y segura~\cite{flyway,liquibase}.

\subsubsection*{Rate limiting}

Los endpoints de registro e inicio de sesión aplican un límite de 5 peticiones por IP
cada 15 minutos mediante el plugin \texttt{@fastify/rate-limit}~\cite{fastify_ratelimit},
configurado con \texttt{global: false} para no afectar al resto de rutas. Las direcciones
de loopback (\texttt{127.0.0.1}, \texttt{::1}) quedan exentas para no interferir con las
pruebas locales.

% ─────────────────────────────────────────────
\subsection{Servicio de recomendación de ruta}
\label{subsec:recomendacion_impl}
% ─────────────────────────────────────────────

El \texttt{recomendacion-service} implementa un algoritmo de recomendación geoespacial
que, dada una ruta entre origen y destino, sugiere las paradas de repostaje óptimas
ponderando el precio del combustible y el desvío necesario para alcanzar cada estación.
Está implementado en \textbf{FastAPI}~\cite{fastapi} con Python, y
combina consultas geoespaciales sobre PostGIS, geometría vectorial con
\textbf{Shapely} y llamadas a la API de \textbf{OpenRouteService}~\cite{openrouteservice}
para el cálculo de rutas y matrices de tiempos.

\subsubsection*{Flujo del algoritmo}

El algoritmo se ejecuta en siete etapas secuenciales:

\begin{enumerate}
  \item \textbf{Obtención de la ruta base.} El servicio solicita a OpenRouteService el
        trazado completo de la ruta entre origen y destino, obteniendo la polilínea geométrica
        y la distancia y duración totales.

  \item \textbf{Prefiltrado de candidatas.} A partir de la polilínea, se construye un
        corredor de búsqueda cuya anchura es \texttt{route\_buffer\_km = min(max\_desvio\_km
          $\times$ 3,\ 50)} km. Este corredor se aplica mediante la función
        \texttt{ST\_DWithin} de \textbf{PostGIS}~\cite{postgis} para recuperar únicamente las
        estaciones cuya geometría cae dentro del área de interés, evitando cargar el catálogo
        completo de más de 11\,000 estaciones. El corredor se refina adicionalmente en memoria
        con \texttt{build\_route\_corridor} de \textbf{Shapely} \cite{shapely}, que construye un buffer
        vectorial preciso alrededor de la polilínea.

  \item \textbf{Cálculo de desvío aproximado.} Para cada candidata se estima un desvío
        preliminar mediante la distancia haversine al punto más cercano de la ruta, implementado
        en \texttt{geo\_math.py}. Este paso descarta rápidamente las estaciones que, aunque
        dentro del corredor geométrico, superan claramente el umbral de desvío configurado.

  \item \textbf{Refinamiento con Matrix API.} El subconjunto de candidatas que supera el
        filtro aproximado se envía a la \textbf{Matrix API de OpenRouteService} \cite{openrouteservice} para obtener
        tiempos de desvío reales en red viaria, que son significativamente más precisos que la
        distancia en línea recta, especialmente en entornos con orografía compleja o vías de
        acceso limitado.

  \item \textbf{Refinamiento exacto con rutas vía parada.} Para las candidatas con mejor
        puntuación preliminar, el servicio solicita a OpenRouteService la ruta completa
        origen\,$\rightarrow$\,estación\,$\rightarrow$\,destino para obtener el sobrecoste
        exacto en tiempo y distancia respecto a la ruta base.

  \item \textbf{Filtrado por límites y puntuación.} Las estaciones que superan los límites
        de desvío configurados (\texttt{max\_desvio\_km}, \texttt{max\_desvio\_min}) se
        descartan. Las restantes se puntúan con la función de \textit{scoring} descrita a
        continuación.
\end{enumerate}

\subsubsection*{Función de \textit{scoring}}

La puntuación de cada estación candidata se calcula mediante una combinación lineal
ponderada de dos factores normalizados:

\begin{equation}
  \text{score} = w_p \cdot (1 - \hat{p}) + w_d \cdot (1 - \hat{d}) + b_{\text{area}}
  \label{eq:scoring}
\end{equation}

donde $\hat{p}$ es el precio normalizado de la estación respecto al rango de precios del
conjunto de candidatas, $\hat{d}$ es el desvío normalizado respecto al desvío máximo
permitido, y $b_{\text{area}} = 0.08$ es un bonus aplicado a estaciones clasificadas como
áreas de servicio, que compensan su habitualmente mayor precio con una mayor accesibilidad
desde la vía principal. Los pesos $w_p$ y $w_d$ son configurables por el usuario en cada
petición y se normalizan automáticamente para que $w_p + w_d = 1$, con valores por defecto
de $0.6$ y $0.4$ respectivamente, priorizando el precio sobre el desvío.

\subsubsection*{Configuración de la petición}

El endpoint \texttt{POST~/api/recomendacion/ruta} acepta los parámetros
\texttt{max\_desvio\_km} y \texttt{max\_desvio\_min} para acotar el espacio de búsqueda,
\texttt{peso\_precio} y \texttt{peso\_desvio} para personalizar la función de
\textit{scoring}, el tipo de combustible del vehículo para filtrar estaciones que
dispongan de ese combustible, y \texttt{top\_n} para controlar el número de
recomendaciones devueltas. Esta flexibilidad permite adaptar el servicio a perfiles de
uso muy diferentes: desde un conductor particular que prioriza el precio mínimo hasta un
transportista profesional que prefiere minimizar el tiempo de desvío.

% ─────────────────────────────────────────────
\subsection{Módulo de predicción de precios}
\label{subsec:prediccion_impl}
% ─────────────────────────────────────────────

El \texttt{prediction-service} implementa un pipeline de machine learning para estimar el
precio de combustible de las estaciones favoritas de los usuarios durante los próximos 7
días. El diseño se orienta a la ejecución periódica en lote (\textit{batch}) más que a la
inferencia en tiempo real, separando físicamente el rol de entrenamiento del rol de
consulta.

\subsubsection*{Selección del modelo}

Se evaluaron tres familias de modelos para la predicción de series temporales de precios
de combustible. Los modelos \textbf{ARIMA} ofrecen buena interpretabilidad y un error
puntual competitivo en series estacionarias a corto plazo, pero presentan dos limitaciones
críticas para este sistema: requieren un modelo independiente por cada par
(estación,~combustible), lo que hace inviable su aplicación sobre las más de
11\,000~estaciones del catálogo nacional, y no incorporan variables exógenas de forma
directa, impidiendo aprovechar la correlación con el precio del crudo Brent. Las redes
\textbf{LSTM} permiten modelar dependencias temporales largas, pero requieren volúmenes
de datos superiores a los disponibles por estación, normalización de las entradas y un
ajuste de hiperparámetros costoso, además, su tiempo de entrenamiento es incompatible con
la ejecución semanal en un entorno serverless con recursos limitados~\cite{hartanto2023lightgbm}.

Se ha optado por \textbf{LightGBM}~\cite{tandfonline_lightgbm} en régimen de
\textbf{regresión cuantílica} por tres razones:

\begin{enumerate}
  \item \textbf{Escalabilidad:} el mismo código entrena modelos para cualquier número de
        estaciones en un único job, sin requerir una instancia independiente por serie.
  \item \textbf{Intervalos de confianza nativos:} la regresión cuantílica con cuantiles
        0.05, 0.50 y 0.95 produce directamente una banda de incertidumbre para cada predicción,
        enriqueciendo la información presentada al usuario sin una etapa adicional de estimación.
  \item \textbf{Rendimiento competitivo:} en la validación sobre datos reales de tres
        estaciones representativas, LightGBM alcanzó un MAPE inferior al 2.3\% en todos los
        casos y superó a ARIMA y XGBoost en las series con mayor variabilidad, donde la
        incorporación de \textit{features} exógenas del Brent aporta mayor
        capacidad predictiva~\cite{joiv_lightgbm}.
\end{enumerate}

\subsubsection*{Ingeniería de \textit{features}}

Para cada par (estación, combustible) se construye una serie temporal diaria con el precio
medio registrado. A partir de ella se generan cuatro grupos de \textit{features}:

\begin{itemize}
  \item \textbf{Features temporales:} día de la semana, indicador de fin de semana, semana
        del año, mes y trimestre.
  \item \textbf{\textit{Lags}:} precios con retardos de 1, 7, 14, 21 y 30 días, que
        capturan la autocorrelación de la serie a distintas escalas.
  \item \textbf{Estadísticos de ventana:} media móvil y desviación típica en ventanas de
        7, 14 y 30 días.
  \item \textbf{Diferencias:} variación de precio a 1 y 7 días, que aproximan la
        derivada de la serie y ayudan al modelo a detectar tendencias recientes.
\end{itemize}

Como variable exógena se incorpora el precio del crudo \textbf{Brent}, obtenido a través
de la API pública de la Reserva Federal de St.\ Louis
(\texttt{FRED:~DCOILBRENTEU}) \cite{fred2026brent}. En caso de que esta fuente no
esté disponible, el servicio recurre a \textbf{yfinance} como fuente alternativa. Los
\textit{lags} del Brent se añaden al vector de \textit{features} aprovechando la
correlación conocida entre el precio del crudo en mercados internacionales y los precios
en surtidor \cite{galeotti2003rockets}.

\subsubsection*{Ciclo de entrenamiento e inferencia}

El pipeline se ejecuta semanalmente mediante un \textbf{Cloud Run Job} disparado por
\textbf{Cloud Scheduler} cada lunes a las 06:00~UTC. En cada ejecución el job realiza los
siguientes pasos: descarga los snapshots Parquet de los últimos 180 días desde Google
Cloud Storage, valida que los datos sean frescos según el umbral \texttt{RAW\_MAX\_AGE\_HOURS},
construye el vector de \textit{features} para cada par (estación, combustible), entrena
tres modelos LightGBM independientes para los cuantiles 0.05, 0.50 y 0.95, genera las
predicciones de los 7 días siguientes de forma secuencial, de forma que cada día
predicho alimenta los \textit{lags} de los días posteriores y persiste los resultados en
la tabla \texttt{prediction\_forecasts} de PostgreSQL con los campos \texttt{precio\_predicho},
\texttt{margen\_min} y \texttt{margen\_max}.

Los artefactos del modelo entrenado (\texttt{.pkl}) se almacenan en el bucket de Google
Cloud Storage bajo el prefijo \texttt{GCS\_MODEL\_PREFIX} para su eventual reutilización
o auditoría. Un backup adicional es configurable mediante \texttt{ENABLE\_GCS\_BACKUP}.

La selección de estaciones a predecir opera en dos modos. En \texttt{SOURCE\_MODE=favorites},
el job consulta la tabla \texttt{user\_favorites} o el endpoint interno del servicio de
usuarios para obtener las estaciones con al menos \texttt{MIN\_FAVORITES\_COUNT} usuarios
que las han marcado como favoritas, limitando el conjunto a las \texttt{TOP\_N\_STATIONS}
más demandadas. Este criterio concentra la capacidad de cómputo en las estaciones con
demanda real, evitando el coste prohibitivo de entrenar modelos para las más de
11\,000 estaciones del catálogo nacional. En \texttt{SOURCE\_MODE=env}, el conjunto se
fija manualmente mediante la variable \texttt{STATION\_IDS}, lo que facilita la ejecución
en entornos de desarrollo o pruebas sin dependencia del servicio de usuarios.

En producción el servicio opera en modo API (\texttt{RUN\_HTTP\_API=true}): no ejecuta el

mínimos al no ejecutar ningún cómputo intensivo durante la petición.

\subsection{Asistente conversacional de voz}
\label{subsec:voice_impl}
% ─────────────────────────────────────────────

El \texttt{voice-assistant-service} implementa un asistente de dominio restringido que
permite al usuario realizar consultas sobre gasolineras en lenguaje natural, tanto por
texto como por voz, y recibir una respuesta en audio. El servicio está implementado con
\textbf{Fastify}~\cite{fastify} y se integra con la API de \textbf{Google Gemini}~\cite{gemini_2_5}
mediante llamadas HTTP REST al cliente \texttt{@google/genai}.

\subsubsection*{Pipeline STT $\rightarrow$ LLM $\rightarrow$ TTS}

El pipeline se ejecuta en tres etapas secuenciales, cada una con su propia llamada a la
API de Gemini. En la etapa de \textbf{reconocimiento de voz} (\textit{Speech-to-Text}),
si la petición incluye audio en base64, el servicio invoca \texttt{generateContent} con
el modelo configurado para transcribir el audio a texto. En la etapa de \textbf{razonamiento}
(\textit{LLM con function calling}), se realiza una primera llamada con las declaraciones
de las funciones disponibles (\texttt{get\_prices} y \texttt{get\_nearest\_stations}), si
el modelo solicita ejecutar alguna de ellas, el servicio las invoca localmente contra el
API Gateway y realiza una segunda llamada con los resultados, obteniendo así la respuesta
final fundamentada en datos reales. En la etapa de \textbf{síntesis de voz}
(\textit{Text-to-Speech}), se realiza una llamada adicional a Gemini con modalidad de
respuesta \texttt{AUDIO}, la respuesta PCM se convierte a WAV y se devuelve en base64
al cliente.

Este diseño con tres llamadas separadas, en lugar de una única llamada multimodal, ofrece
mayor control sobre cada etapa y facilita el manejo de errores y los fallbacks por modelo.

\subsubsection*{Guardarraíles y control del dominio}

Dado que el asistente está diseñado para un dominio muy específico, se implementan varios
mecanismos para evitar un uso fuera de ámbito. Un \textbf{comprobador de alcance}
(\textit{scope guard}) basado en palabras clave evalúa la transcripción antes de invocar
el LLM, si la consulta no contiene términos relacionados con combustible, gasolineras,
precios o navegación, y la variable \texttt{VOICE\_GUARDRAILS\_ENABLED} está activa, el
servicio devuelve una respuesta predefinida sin consumir tokens del modelo. El modo
\texttt{strict} rechaza directamente la petición, mientras que el modo \texttt{soft} permite continuar
con una advertencia al modelo.

Adicionalmente, el \textbf{system prompt} construido por \texttt{buildSystemInstruction}
instruye explícitamente al modelo sobre el idioma, el tono y el formato esperados, y lo
obliga a utilizar las funciones declaradas para cualquier consulta de datos, reduciendo el
riesgo de alucinaciones. El function calling se ejecuta en el servidor, de forma que el
LLM nunca interactúa directamente con el gateway, sino que proporciona los parámetros de
la llamada y espera los resultados.

Otros controles operativos incluyen límites de tamaño de payload
(\texttt{VOICE\_MAX\_TEXT\_CHARS}, \\ \texttt{VOICE\_MAX\_AUDIO\_BASE64\_CHARS}), timeouts
por llamada a Gemini con \texttt{AbortController}, y rate limiting por IP para el
transporte WebSocket. Como limitación conocida, el rate limiter no se aplica actualmente
al endpoint HTTP \texttt{/voice/dialog}, aspecto identificado para una iteración futura.

\subsubsection*{Nota sobre el transporte WebSocket}

El servicio implementa tanto el endpoint HTTP \texttt{POST /voice/dialog} como un
endpoint WebSocket \texttt{/ws/voice} para streaming de audio. Durante el desarrollo se
identificaron problemas de estabilidad de conexión con el transporte WebSocket en el
entorno de Cloud Run, asociados al comportamiento de las peticiones de larga duración en
infraestructura serverless. Por ello, el cliente frontend utiliza actualmente el endpoint
HTTP, que ofrece un comportamiento predecible en todos los entornos de despliegue.

% ─────────────────────────────────────────────
\subsection{Frontend: React PWA con Nginx}
\label{subsec:frontend_impl}
% ─────────────────────────────────────────────

El frontend es una \textit{Single Page Application} (SPA) construida con
\textbf{React 19}~\cite{react} y \textbf{TypeScript}~\cite{typescript}, compilada con
\textbf{Vite}~\cite{vite} y empaquetada como \textbf{Progressive Web App}
(PWA)~\cite{pwa_spec}. En producción, los artefactos estáticos generados por Vite se
sirven mediante \textbf{Nginx}~\cite{nginx} dentro de un contenedor Docker de imagen
mínima.

\subsubsection*{Configuración de Nginx}

Nginx está configurado para servir el directorio \texttt{dist} generado por Vite desde
\texttt{/usr/share/nginx/html}. La directiva \texttt{try\_files \$uri \$uri/ /index.html}
garantiza que todas las rutas no estáticas sean resueltas por el router de React en
cliente, comportamiento necesario para el correcto funcionamiento de la navegación SPA.
Los assets estáticos con huella de contenido (\texttt{.js}, \texttt{.css}, fuentes e
imágenes) se cachean durante un año con la cabecera
\texttt{Cache-Control: public, immutable}, lo que maximiza el rendimiento en visitas
recurrentes. Nginx no actúa como proxy inverso para la API en producción: todas las
llamadas al backend se dirigen directamente a la URL del API Gateway configurada en
tiempo de compilación mediante la variable \texttt{VITE\_API\_BASE\_URL}.

\subsubsection*{Progressive Web App}

La aplicación implementa el estándar PWA mediante el plugin
\textbf{vite-plugin-pwa}~\cite{vite_plugin_pwa} con Workbox~\cite{workbox} como gestor del
Service Worker. El fichero \texttt{manifest.webmanifest} define los metadatos necesarios
para la instalación en dispositivos móviles (nombre, iconos, color de tema y modo de
visualización \textit{standalone}). El Service Worker implementa una estrategia de caché
que permite el acceso a la interfaz aunque la conexión sea intermitente, aunque las
consultas de datos en tiempo real requieren conectividad activa.

\subsubsection*{Internacionalización y diseño responsivo}

La interfaz soporta tres idiomas (español, inglés y euskera) mediante
\textbf{i18next}~\cite{i18next} con detección automática del idioma del navegador. El
diseño sigue un enfoque \textit{mobile-first} con \textbf{TailwindCSS}~\cite{tailwind},
garantizando que la experiencia sea funcional tanto en pantallas de escritorio como en
dispositivos móviles, donde el caso de uso de búsqueda de gasolineras en ruta es
predominante. El mapa interactivo se implementa con \textbf{Leaflet}~\cite{leaflet} a
través de la librería \texttt{react-leaflet}, con soporte para clustering de marcadores
a niveles de zoom bajos para evitar la saturación visual cuando hay muchas estaciones en
el área visible.

% ─────────────────────────────────────────────
\subsection{Despliegue en GCP y pipeline CI/CD}
\label{subsec:despliegue_impl}
% ─────────────────────────────────────────────

Todos los microservicios se despliegan en \textbf{Google Cloud Run}~\cite{cloud_run2026},
plataforma serverless que gestiona el escalado automático en función de la demanda y
factura únicamente por el tiempo de cómputo efectivo. Esta elección elimina la necesidad
de gestionar instancias o clústeres, lo que reduce significativamente la carga operacional
para un proyecto académico sin comprometer la disponibilidad.

\subsubsection*{Pipeline de CI/CD}

Cada microservicio dispone de su propio fichero \texttt{cloudbuild-*.yaml} que define un
pipeline de \textbf{Cloud Build}~\cite{cloud_build2026} independiente. Los pipelines se
activan automáticamente con cada \textit{push} a la rama \texttt{main} del repositorio.
El proceso consta de tres pasos: construcción de la imagen Docker, publicación en
\textbf{Artifact Registry}~\cite{artifact_registry} (región \texttt{europe-west1}), y
despliegue en Cloud Run con las variables de entorno inyectadas desde
\textbf{Secret Manager}~\cite{secret_manager}. Este diseño garantiza que ningún secreto
esté presente en el repositorio ni en los logs de compilación.

El pipeline del frontend incluye validaciones adicionales en el paso de compilación: la
variable \texttt{VITE\_API\_BASE\_URL} debe apuntar a un dominio HTTPS (no a
\texttt{localhost}), lo que previene despliegues accidentales de configuraciones de
desarrollo.

\subsubsection*{Comunicación entre servicios en Cloud Run}

En Cloud Run cada servicio tiene una URL pública asignada por Google, pero la comunicación
interna entre servicios no debe transitar por internet. Para ello, el gateway utiliza
tokens IAM de servicio (\textit{service account tokens}) inyectados mediante el módulo
\texttt{cloudRunAuthFetch.js}, que autentica cada llamada inter-servicio ante la
infraestructura de Google sin exponer los servicios internos a peticiones externas no
autorizadas.

\subsubsection*{Base de datos y almacenamiento}

La base de datos PostgreSQL reside en \textbf{Neon Tech}~\cite{neon_tech2026}, proveedor de
PostgreSQL serverless que ofrece \textit{branching} de bases de datos, hibernación
automática de instancias inactivas y conexión mediante cadena estándar
\texttt{postgresql://}. La elección de un proveedor externo en lugar de Cloud SQL de GCP
responde a la disponibilidad de un plan gratuito suficiente para las cargas del proyecto
y a la independencia del proveedor cloud para la capa de datos.

Los snapshots Parquet del pipeline de ML y los modelos entrenados se almacenan en
\textbf{Google Cloud Storage}~\cite{cloud_storage2026}, que actúa como capa de almacenamiento de objetos
entre el servicio de gasolineras (productor de datos), el job de predicción (consumidor y
productor de modelos) y el servicio de predicción en modo API (lector de resultados).

\subsection{Seguridad transversal}
\label{subsec:seguridad_impl}
% ─────────────────────────────────────────────

Además de los mecanismos de autenticación descritos en la sección anterior, el sistema
implementa un conjunto de controles de seguridad aplicados de forma transversal.


La comunicación entre el gateway y los microservicios internos está protegida por la
cabecera \texttt{X-Internal-Secret}: los endpoints de administración (sincronización
forzada, export, consulta de favoritos globales) rechazan con \texttt{403 Forbidden}
cualquier petición que no incluya el secreto configurado en \texttt{INTERNAL\_API\_SECRET}.
Este mecanismo evita que dichos endpoints sean accesibles desde el exterior incluso si
un servicio queda expuesto inadvertidamente.

La validación de datos de entrada se realiza en capa de aplicación con
\textbf{Pydantic v2}~\cite{pydantic} en los servicios Python y con los esquemas de
validación de Fastify en los servicios Node.js. Los parámetros de consulta con potencial
de abuso (radio de búsqueda, límite de resultados) tienen cotas máximas declaradas
explícitamente. Las consultas a PostgreSQL utilizan siempre parámetros preparados, sin
concatenación de cadenas, eliminando el vector de inyección SQL~\cite{owasp_sqli}.

Además, la política de seguridad transversal incluye comprobaciones y controles operativos
que se aplican a todos los servicios: verificación de que los endpoints internos requieran
la cabecera \texttt{X-Internal-Secret}, activación de cabeceras de seguridad mediante
\textbf{Helmet.js} en los servicios Node.js, saneamiento de logs para evitar el registro
de información sensible y aplicación coherente de límites de tasa y reglas de perímetro
(Cloudflare / reglas del gateway). Estas comprobaciones reducen la superficie de ataque
y facilitan la auditoría de seguridad.

\section{Plan de Pruebas}
\label{sec:plan-pruebas}

La estrategia de pruebas de TankGo prioriza la verificación de la lógica de negocio crítica y
la integración entre componentes, dentro de las restricciones temporales propias de un proyecto
académico individual. Se ha adoptado un enfoque \textit{test-last}: las pruebas se escriben
una vez estabilizada la implementación, lo que resulta más adecuado cuando los requisitos
evolucionan durante el desarrollo~\cite{sommerville2011}. La ejecución se delega íntegramente a pipelines 
de integración continua en GitHub
Actions~\cite{github_actions}, de modo que los resultados son reproducibles y auditables
sin depender del entorno local de desarrollo.

La cobertura se mide únicamente sobre lógica de negocio, excluyendo código de infraestructura
(conexiones a base de datos, configuración de arranque) mediante ficheros \texttt{.coveragerc}.
Esto evita inflar artificialmente el porcentaje con código trivial y refleja con mayor fidelidad
la calidad real de las pruebas.

\section{Incidencias del Proyecto}
\label{sec:incidencias}
% ═════════════════════════════════════════════

Esta sección documenta los problemas técnicos más relevantes encontrados a lo largo del
desarrollo de TankGo, las causas identificadas y las soluciones adoptadas. Lejos de ser
una lista de fallos, las incidencias recogidas aquí reflejan el proceso de toma de
decisiones técnicas ante situaciones no previstas en la planificación inicial y constituyen
parte del aprendizaje más valioso del proyecto.

% ─────────────────────────────────────────────
\subsection{Integración con servicios externos y APIs de terceros}
\label{subsec:inc_externos}
% ─────────────────────────────────────────────

\subsubsection*{Renderizado del mapa con todas las gasolineras activo}

Inicialmente el frontend pintaba en el mapa todas las gasolineras del
catálogo nacional simultáneamente al cargar la vista, lo que generaba un retraso notable
en la respuesta del navegador y una experiencia visualmente saturada con más de 11\,000
marcadores activos. El problema no solo era de red, sino de rendering en el hilo principal
del navegador.

La solución que se ha abordado son dos enfoques. En el backend, las consultas de gasolineras se
limitaron geoespacialmente mediante PostGIS~\cite{postgis}, devolviendo las
estaciones dentro del radio de búsqueda del usuario en lugar del catálogo completo. En el
frontend, se ha integrado \textbf{Leaflet MarkerCluster}~\cite{leaflet_markercluster} para
agrupar los marcadores en función del nivel de zoom, de modo que a escalas reducidas el
mapa muestra agregados en lugar de marcadores individuales. Esta combinación redujo el
tiempo de renderizado inicial a valores aceptables y ha mejorado la
usabilidad en dispositivos móviles.

\subsubsection*{Algoritmo de recomendación: estaciones cercanas y cálculo de desvíos}

El servicio de recomendación de ruta ha presentado varias dificultades durante su
implementación. El primer problema ha sido la definición de \textit{cercanía} a una ruta: una
búsqueda por radio desde el punto de origen no tiene sentido para rutas largas, ya que
excluye las estaciones ubicadas a lo largo del trayecto. Se ha adoptado un enfoque basado en
un corredor geométrico alrededor de la polilínea de la ruta, implementado con
\textbf{PostGIS}~\cite{postgis} (\texttt{ST\_DWithin} sobre la geometría de la ruta) y
refinado con la librería \textbf{Shapely} para el cálculo de buffers vectoriales.

El segundo problema ha sido el cálculo del tiempo de desvío real para cada estación candidata.
Las primeras versiones calculaban el desvío como la distancia haversine al punto más
cercano de la ruta, pero este valor no correlaciona bien con el tiempo de desvío en
carretera, que depende de la red viaria real. La solución ha sido utilizar la
\textbf{Matrix API de OpenRouteService}~\cite{gisops_routing_overview} para obtener
tiempos de desvío reales, refinando posteriormente los resultados mediante llamadas
individuales a la Directions API para los candidatos con mejor puntuación.

Adicionalmente, la evasión de peajes solicitada como parámetro de la ruta no siempre
producía buenos resultados,
debido a que la API de Matrix de OpenRouteService no tiene en cuenta esta variable
en el cálculo de tiempos de desvío,
lo que llevaba a recomendar estaciones que técnicamente cumplían el criterio
de cercanía pero que implicaban ir por carreteras de pago. Sin embargo, la llamada para
marcar el recorrido si que permite evitar peajes.
Esta limitación quedó documentada como restricción conocida del sistema
(véase la sección~\ref{sec:alcance}).

% ─────────────────────────────────────────────
\subsection{Infraestructura y despliegue en Google Cloud Platform}
\label{subsec:inc_gcp}
% ─────────────────────────────────────────────

\subsubsection*{Configuración de variables de entorno y Secret Manager}

Una de las primeras incidencias en el despliegue fue la gestión de variables de entorno
en Cloud Run. Inicialmente se ha intentado configurar algunas variables directamente en los
ficheros \texttt{cloudbuild.yaml}, lo que presentaba dos problemas: los valores quedaban
expuestos en los logs de compilación de Cloud Build, y cualquier cambio de valor requería
una nueva ejecución del pipeline.

La solución fue migrar todos los secretos a \textbf{Secret Manager}~\cite{secret_manager}
y referenciarlos en los ficheros de despliegue mediante la sintaxis
\texttt{secretKeyRef}, de forma que Cloud Run los inyecta en tiempo de ejecución sin que
el valor nunca aparezca en el código ni en los logs. La distinción entre variables de
configuración no sensibles (inyectadas directamente como \texttt{env} en el
\texttt{cloudbuild.yaml}) y secretos (gestionados por Secret Manager) quedó establecida
como convención del proyecto.

\subsubsection*{Conflicto de DNS entre Cloudflare y GCP}

El proyecto utiliza Cloudflare como proveedor de DNS para el dominio público de TankGo.
Durante la configuración inicial surgió un conflicto entre la gestión de certificados TLS
de Cloud Run (que espera verificar la propiedad del dominio mediante registros DNS
específicos) y el modo \textit{proxy} activo de Cloudflare, que oculta las IPs reales
del servidor y puede interferir con el proceso de validación de dominio de Google.

La resolución ha requerido desactivar temporalmente el modo proxy de Cloudflare durante la
verificación de dominio en Cloud Run, para posteriormente reactivarlo una vez completada
la validación.

\subsubsection*{Problemas de conexión SSL con Neon Tech}

La base de datos PostgreSQL en \textbf{Neon Tech}~\cite{neon_tech2026} requiere conexión
cifrada SSL obligatoria. Durante las primeras pruebas de despliegue en Cloud Run, algunos
microservicios fallaban al establecer la conexión con errores relacionados con la
verificación del certificado TLS. El problema tenía dos causas: la cadena de conexión no
incluía el parámetro \texttt{sslmode=require} de forma explícita, y la imagen base de
algunos servicios no incluía los certificados raíz de CA necesarios para verificar el
certificado del servidor Neon.

La solución ha consistido en estandarizar la cadena de conexión en todos los servicios para
incluir \texttt{?sslmode=require} y asegurarse de que las imágenes Docker incluyeran el
paquete \texttt{ca-certificates}, disponible en todas las distribuciones base utilizadas.

% ─────────────────────────────────────────────
\subsection{Seguridad: ataques detectados y medidas adoptadas}
\label{subsec:inc_seguridad}
% ─────────────────────────────────────────────

\subsubsection*{Intentos de explotación detectados en producción}

Poco después de publicar el sistema en producción con dominio público, el análisis de logs
de Cloud Run ha revelado un volumen de peticiones automatizadas intentando acceder
a rutas no existentes en el proyecto, pero características de vulnerabilidades conocidas en
otras plataformas: rutas como \texttt{/wp-admin}, \texttt{/phpMyAdmin},
\texttt{/.env} o \texttt{/admin/config.php}. Este tipo de escaneo automatizado
es habitual en cualquier servidor público expuesto a internet y no representa una amenaza
directa para TankGo, cuya API no implementa ninguno de esos patrones, pero confirmó la
necesidad de reforzar la configuración de seguridad perimetral.

Como respuesta, se han configurado reglas en \textbf{Cloudflare} para bloquear peticiones
dirigidas a patrones de rutas asociados a exploits conocidos de WordPress, PHP y paneles
de administración genéricos, así como para limitar la tasa de peticiones desde IPs que
generaran un número anómalo de errores \texttt{404} en un intervalo corto. Además, se
incorporaron reglas a nivel DNS y de firewall específicas para mitigar vectores dirigidos
a componentes PHP (por ejemplo, detección de rutas y extensiones típicas), reduciendo la
superficie de ataque sobre ese tipo de recursos. Se ha añadido también un análisis de
cabeceras y patrones de petición HTTP/1.1 para identificar clientes automatizados, para
peticiones con comportamiento sospechoso el perímetro puede responder con un reto de
verificación (challenge), por ejemplo comprobaciones JavaScript o CAPTCHA, antes de
permitir el acceso, lo que ayuda a filtrar bots y ahorrar peticiones fraudulentas.
Las reglas de \textit{rate limiting} del gateway se revisaron y ajustaron para garantizar que los
endpoints de mayor exposición (autenticación, consulta de precios) tuviesen límites
apropiados.

% ─────────────────────────────────────────────
\subsection{Asistente conversacional: transporte y control del dominio}
\label{subsec:inc_voice}
% ─────────────────────────────────────────────

\subsubsection*{Abandono del transporte WebSocket en favor de HTTP}

El diseño inicial del asistente de voz contemplaba una conexión \textbf{WebSocket}
persistente entre el cliente y el servicio, con el objetivo de permitir streaming de
audio en tiempo real y reducir la latencia percibida. Durante el desarrollo se implementó
el endpoint \texttt{/ws/voice} y se probó su funcionamiento en entorno local, donde se
comportaba correctamente.

Sin embargo, al desplegar el servicio en \textbf{Cloud Run}, surgieron problemas de
estabilidad que impedían establecer la conexión WebSocket de forma consistente. La causa
raíz está relacionada con el comportamiento de Cloud Run ante conexiones de larga
duración: la infraestructura serverless no garantiza la persistencia de conexiones
WebSocket bajo condiciones de escalado o reasignación de instancias, y los timeouts de
la plataforma interferían con el ciclo de vida esperado de la conexión.

Ante esta situación, se ha tomado la decisión de migrar el transporte del cliente al endpoint
HTTP \texttt{POST /voice/dialog}, que encapsula el ciclo completo STT\,$\rightarrow$\,LLM\,$\rightarrow$\,TTS
en una única petición request-response. Esta decisión sacrifica el streaming de audio
parcial pero ofrece un comportamiento completamente predecible en todos los entornos de
despliegue y elimina la complejidad de gestión del estado de conexión en el cliente.
El endpoint WebSocket se mantiene implementado en el servidor pero no se utiliza en
producción.

\subsubsection*{Diseño de guardarraíles para el control del dominio}

Durante las pruebas del asistente se ha detectado que el modelo de lenguaje, sin restricciones
explícitas, tendía a responder consultas fuera del dominio de TankGo (preguntas generales,
conversación casual, solicitudes de información no relacionada con combustible o
gasolineras). Esto no era deseable en un asistente diseñado para uso en conducción, donde
las respuestas deben ser breves, directas y estrictamente relacionadas con la tarea.

Se ha diseñado un sistema de guardarraíles en dos niveles. El primer nivel es un
\textbf{comprobador de alcance} previo al LLM que evalúa la presencia de términos del
dominio en la transcripción, si no se detecta ninguno, el servicio devuelve directamente
una respuesta predefinida sin consumir tokens del modelo. El segundo nivel actúa mediante
el \textbf{system prompt}: las instrucciones del sistema definen explícitamente el papel
del asistente, los temas permitidos, el formato de respuesta esperado y la obligación de
utilizar las funciones declaradas (\textit{tool calling}) para cualquier consulta de datos
en lugar de generar respuestas basadas en conocimiento interno del modelo.

La combinación de ambos niveles ha reducido de forma efectiva las respuestas fuera de dominio,
aunque la calibración del comprobador de alcance requirió varias iteraciones para encontrar
un equilibrio entre el rechazo de consultas legítimas con vocabulario informal y la
permisividad excesiva ante consultas claramente fuera de ámbito.
